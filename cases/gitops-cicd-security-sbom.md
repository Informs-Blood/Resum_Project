# Кейс: GitOps CI/CD с полным security-стеком и SBOM

Построение пайплайна, в котором сборка и тесты выполняются в CI (GitHub Actions / Jenkins), артефакты подписываются и сопровождаются SBOM, деплой идёт через GitOps (Argo CD), а в кластере действуют политики по образам и манифестам (Conftest/OPA, Kyverno). Дополнительно — сканирование на уязвимости, зависимостей и compliance.

---

## Задача

Требовалось разделить зоны ответственности: **CI** — сборка, тесты, безопасность артефактов; **CD** — декларативный деплой из Git без ручного kubectl apply. Нужны были: генерация и проверка SBOM (Software Bill of Materials), подпись образов для доверия в supply chain, проверка манифестов и образов по политикам до попадания в кластер, а также единая точка правды для окружений (staging/prod) в Git.

---

## Решение

### Архитектура

- **Исходный код** в Git (GitLab/GitHub); при push/merge — запуск CI (GitHub Actions или Jenkins / Tekton).  
- **CI-пайплайн**: сборка образа → тесты → сканирование (Trivy, Snyk) → генерация SBOM (Syft) → подпись образа (Cosign) → push в registry с артефактами (SBOM, подпись).  
- **Отдельный Git-репозиторий (или ветка)** с манифестами деплоя: Helm charts или raw YAML, разложенные по окружениям (например `environments/staging`, `environments/prod`).  
- **Argo CD** (или Flux) подписан на этот репозиторий и синхронизирует состояние кластера с Git; деплой = коммит в репо манифестов.  
- **Политики**: проверка манифестов (Conftest/OPA) и/или admission-контроллер в кластере (Kyverno) — запрет неподписанных образов, образов с критичными CVE, несоответствия стандартам (labels, resource limits).

### Этапы CI (сборка и безопасность)

1. **Build** — сборка Docker-образа (Docker Buildx / Kaniko в K8s), теги по коммиту и ветке.  
2. **Test** — юнит- и интеграционные тесты, при необходимости e2e в отдельном job.  
3. **Vulnerability scanning** — **Trivy** (image + при необходимости fs): блокировка при critical/high по политике. **Snyk** или **Dependabot** — зависимости (SCA).  
4. **SBOM** — **Syft** генерирует SBOM в формате SPDX или CycloneDX; артефакт сохраняется и при желании пушится в registry вместе с образом.  
5. **Подпись образа** — **Cosign** (Sigstore): подпись образа и при необходимости SBOM; в registry хранятся образ + подпись.  
6. **Push** — образ (с подписью и опционально SBOM) отправляется в **Harbor** или другой registry с поддержкой OCI.

### Этапы CD (GitOps и политики)

1. **Репозиторий манифестов** — изменения в Helm values или K8s YAML коммитятся (вручную или из CI при успешном прохождении пайплайна — обновление тега образа).  
2. **Argo CD** — следит за репозиторием и применяет изменения в кластере; авто-синхронизация или ручное Sync; health и статус в UI.  
3. **Проверка манифестов до применения** (опционально в CI для репо манифестов): **Conftest** с регополитиками (OPA) — проверка resource limits, запрет latest, запрет hostPath и т.д.; пайплайн в репо манифестов падает при нарушении.  
4. **Admission в кластере** — **Kyverno** (или OPA Gatekeeper): политики вида «только образы из нашего registry», «образы должны быть подписаны Cosign», «запрет образов с критичными CVE» (интеграция с Trivy/Kyverno). Блокировка неподходящих Pod/Deployment на уровне API server.

### Дополнительно

- **Уведомления**: Argo CD и CI шлют события в Slack/Teams при деплое или падении синхронизации.  
- **Compliance/аудит**: SBOM и результаты сканирований хранятся; при необходимости экспорт в системы compliance (например для регуляторов или внутреннего аудита).

---

## Инструменты

| Категория | Инструменты |
|-----------|-------------|
| **CI** | GitHub Actions, Jenkins и/или Tekton (Kubernetes-native) |
| **Сборка** | Docker Buildx, Kaniko |
| **Сканирование** | Trivy (image, fs, SBOM), Snyk, Dependabot |
| **SBOM** | Syft (SPDX, CycloneDX) |
| **Подпись и supply chain** | Cosign (Sigstore) |
| **Реестр** | Harbor (или GitLab Registry, ECR, GCR) с OCI |
| **GitOps** | Argo CD (или Flux) |
| **Манифесты** | Helm, Kustomize, raw YAML |
| **Policy as Code** | Conftest, Open Policy Agent (OPA), Kyverno |
| **Кластер** | Kubernetes |

---

## Результат

- Чёткое разделение: CI отвечает за артефакты и их безопасность; CD — за применение только того, что описано в Git, с видимостью и откатом через Git history.  
- Supply chain: образы подписаны, SBOM есть — можно проверять состав и соответствие политикам; при необходимости — блокировка неподписанных или уязвимых образов в кластере (Kyverno).  
- Один источник правды для окружений (Git), предсказуемый деплой и откат через revert коммита.  
- Соответствие современным практикам GitOps и DevSecOps с акцентом на SBOM и подпись артефактов.
