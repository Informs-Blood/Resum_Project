# Кейс: Полноценный DevSecOps-пайплайн на GitLab CI

Внедрение сквозного CI/CD с встроенной безопасностью: от коммита до продакшена — линтинг, тесты, SAST/DAST, сканирование зависимостей и образов, проверка IaC и секретов, качество кода и контролируемый деплой.

---

## Задача

Нужно было выстроить единый пайплайн сборки и выката приложений так, чтобы на каждом этапе выполнялись проверки безопасности и качества без ручного запуска. Требования: блокировка коммитов с утёкшими секретами, проверка инфраструктурного кода (Terraform), статический и динамический анализ приложения, сканирование Docker-образов и зависимостей, gate по качеству кода и прозрачный процесс деплоя в staging и prod.

---

## Решение

Пайплайн реализован в **GitLab CI** (`.gitlab-ci.yml`), стадии идут последовательно с возможностью параллелизма внутри стадий. При падении любой проверки сборка не доходит до деплоя.

### Последовательность стадий

1. **Pre-check (секреты и базовая валидация)**  
   - **Gitleaks** — сканирование репозитория на утёкшие секреты (ключи API, пароли, токены). При обнаружении — fail пайплайна, коммит не мержится.  
   - Опционально **TruffleHog** или встроенный GitLab Secret Detection для дублирования проверки.

2. **Lint**  
   - **Hadolint** — проверка Dockerfile на лучшие практики и типичные уязвимости (USER non-root, многослойность, кэш).  
   - Линтеры кода по стеку: **ESLint** (JS/TS), **mypy** / **Pylint** (Python), **golangci-lint** (Go) — в зависимости от репозитория.  
   - **Checkov** — сканирование Terraform/CloudFormation/Kubernetes манифестов (IaC) на небезопасные настройки (открытые порты, отсутствие лимитов, публичные бакеты и т.п.).

3. **Unit- и интеграционные тесты**  
   - Запуск тестов (pytest, jest, go test) с замером покрытия.  
   - **SonarQube** (или GitLab Code Quality) — анализ кода на дубли, запахи, уязвимости; quality gate: при падении ниже порога (например по покрытию или по критичным issues) стадия падает.

4. **SAST (Static Application Security Testing)**  
   - **Semgrep** — правила для типичных уязвимостей (injection, XSS, небезопасное использование крипто и т.д.) по нескольким языкам.  
   - **Bandit** — для Python-кода (хардкод паролей, небезопасные функции).  
   - Результаты передаются в GitLab Security Dashboard (SAST report), при критичных находках — политика fail.

5. **Build и сканирование образов**  
   - Сборка Docker-образа через **Docker Buildx** (multi-stage, кэш слоёв).  
   - **Trivy** — сканирование образа на уязвимости в пакетах (OS + зависимости) и на небезопасные настройки (Dockerfile). Политика: блокировка при наличии критичных CVE или заданный порог.  
   - Пуш образа только при прохождении Trivy в **Harbor** (или GitLab Container Registry) с тегом по коммиту/ветке.

6. **SCA (Software Composition Analysis)**  
   - Сканирование зависимостей (npm, pip, go.mod) — **Trivy fs** или **Snyk** / **Dependabot** — выявление известных уязвимостей в библиотеках. Отчёт в пайплайне и при критичных — fail.

7. **Deploy staging**  
   - Выкат в staging через **Helm** в тестовый кластер Kubernetes (или через Ansible/другой деплой-инструмент).  
   - После деплоя — smoke-тесты и проверка доступности сервиса.

8. **DAST (Dynamic Application Security Testing)**  
   - **OWASP ZAP** (baseline или full scan) против развёрнутого staging-окружения: проверка типичных веб-уязвимостей (SQLi, XSS, небезопасные заголовки).  
   - При критичных находках — пайплайн не переходит в prod.

9. **Deploy production**  
   - Ручной gate (manual job в GitLab) или по тегу/релизу.  
   - Деплой в prod тем же способом (Helm/Argo CD), с опциональной интеграцией **Argo CD** для синхронизации из Git (манифесты/Helm values в отдельном репозитории).

### Дополнительно

- **Артефакты**: отчёты Trivy, SonarQube, Semgrep, Gitleaks сохраняются как артефакты GitLab и отображаются в Security / Code Quality.  
- **Уведомления**: при падении пайплайна или при деплое — уведомления в Slack/Telegram/Teams.  
- **Политики**: в репозитории заданы правила — например «не мержить при открытых critical/high в Trivy или при провале SonarQube gate».

## Инструменты

| Категория | Инструменты |
|-----------|-------------|
| **Оркестрация пайплайна** | GitLab CI/CD |
| **Секреты** | Gitleaks, GitLab Secret Detection, TruffleHog (опц.) |
| **Lint** | Hadolint (Dockerfile), ESLint, mypy, Pylint, golangci-lint |
| **IaC** | Checkov (Terraform, K8s, CloudFormation) |
| **Тесты и качество кода** | pytest / jest / go test, SonarQube, GitLab Code Quality |
| **SAST** | Semgrep, Bandit (Python) |
| **Сборка и образы** | Docker Buildx, Trivy (image + filesystem) |
| **SCA** | Trivy, Snyk / Dependabot |
| **DAST** | OWASP ZAP |
| **Реестр и деплой** | Harbor / GitLab Container Registry, Helm, Kubernetes, Argo CD (опц.) |
| **Уведомления** | Slack / Telegram / Teams, GitLab integrations |

---

## Результат

- Единый пайплайн: от push до prod с явными security- и quality-gates на каждом этапе.  
- Раннее выявление секретов, уязвимостей в коде и зависимостях, небезопасных настроек IaC и образов до выката.  
- Прозрачность: все отчёты в GitLab (Security Dashboard, Code Quality, артефакты), понятные правила блокировки мержа и деплоя.  
- Соответствие практикам shift-left и DevSecOps: безопасность встроена в процесс, а не отдельные разовые проверки.
