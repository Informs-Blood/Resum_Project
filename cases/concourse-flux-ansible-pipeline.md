# Кейс: CI/CD на Concourse с Flux, Ansible и альтернативной сетевой плоскостью

Внедрение и поддержка пайплайнов на базе Concourse CI в **Kubernetes**: конфигурация пайплайнов в Git (GitOps для самих пайплайнов), запуск Ansible-плейбуков из Concourse для инфраструктуры и конфигурации нод, сборка образов и Helm-чартов с публикацией в Harbor, деплой в Kubernetes через Flux (GitOps). Платформа развёрнута **в нескольких ЦОДах** для отказоустойчивости; используется топология **Master/Slave** (control plane и воркеры, плюс репликация бэкенда Concourse). В кластере вместо стандартного kube-proxy — CNI на базе eBPF (Cilium).

---

## Задача

Требовалось выстроить единую цепочку: **сборка и тесты** в CI, **конфигурация серверов и инфраструктуры** через автоматизацию без ручного запуска, **деплой приложений** декларативно из Git без прямого `kubectl apply`. Вся платформа должна работать **на Kubernetes**, быть разнесена по **нескольким ЦОДам** для отказоустойчивости и использовать конфигурацию **Master/Slave** (управляющие и рабочие компоненты, репликация состояния). Дополнительно — единая сетевая плоскость в кластере с возможностью замены стандартного kube-proxy на решение с eBPF.

---

## Решение

### Развёртывание на Kubernetes и отказоустойчивость

- **Вся платформа на Kubernetes**: Concourse (web/ATC и воркеры), Flux, Harbor, вспомогательные сервисы (S3, очереди и т.д.) — работают в кластере как поды/деплойменты, управляются через Helm или манифесты из Git.
- **Размещение в нескольких ЦОДах**: ноды кластера и компоненты CI/CD разнесены по разным дата-центрам; при сбое одного ЦОД остаются рабочие инстансы в других. Маршрутизация и балансировка трафика на Concourse web и воркеры учитывают топологию (node affinity, topology spread).
- **Конфигурация Master/Slave**:
  - **Control plane и воркеры**: узлы управления (master) — etcd, API server, scheduler, controller-manager; воркеры (worker/slave) — пулы нод для выполнения задач. Воркеры могут быть выделены по командам или типам джоб (например тяжёлые сборки — отдельный пул).
  - **Бэкенд Concourse**: PostgreSQL в режиме primary + standby (master/slave) для хранения состояния пайплайнов и очередей; при падении primary — переключение на standby (ручное или через Patroni/оператор). Так CI сохраняет работоспособность при сбое БД в одном ЦОД.
- Конфигурация Master/Slave и размещение по ЦОДам задаётся в манифестах и values Helm; Flux синхронизирует эти настройки при изменении репозитория.

### Роль Concourse

- **Оркестратор CI**: все пайплайны описаны в Git; при изменении репозитория родительский пайплайн через `set_pipeline` обновляет дочерние (GitOps для пайплайнов). Управление через **fly CLI** (`fly set-pipeline`, `fly intercept` и т.д.).
- **Ресурсы (resources)**: Git (исходники и репо с конфигом Concourse), **Semver** в S3 для версий артефактов, GitLab Merge Request (триггер и обновление статуса MR), **Helm** (шаблонизация, package, push), **Kubernetes** (при необходимости — деплой/проверки в кластер), **Telegram** (уведомления об успехе/провале джоб).
- **Задачи (tasks)**: линтинг кода и Dockerfile, сборка OCI-образов (multi-stage), тесты (в т.ч. установка Helm-чарта в тестовый namespace и прогон тестов), обогащение версии, публикация образа в **Harbor**.
- **Команды (teams)** разнесены по продуктам/направлениям (например itrack, infra, datascan); у каждого — свои пайплайны и переменные из `envs.yaml`.

### Связка Concourse и Ansible

- В репозитории с пайплайнами хранятся **инвентарь Ansible** (по группам: concourse, nodes, gitlab, и т.д.) и плейбуки (обновление нод, настройка containerd, увеличение лимитов, разовое применение конфигурации).
- Concourse запускает Ansible через **resource type** (например `concourse-ansible-playbook-resource`): job делает `get` репо, `put` в ресурс ansible-playbook с указанием inventory, playbook и `vars_file` из того же репо.
- Так из Concourse по расписанию или по триггеру выполняются плейбуки на реальные хосты (concourse-воркеры, ноды K8s, прочие серверы) без отдельного Jenkins или ручного запуска.

### Сборка образов и Helm

- Пайплайн приложения: `get` кода → линтеры (язык + Hadolint для Dockerfile) → **OCI build** (образ по Dockerfile) → тесты (установка чарта в кластер, smoke) → пуш образа в Harbor с тегом.
- Отдельные пайплайны под **Helm-чарты**: шаблонизация (helm template), линтинг чарта, package, push в OCI-реестр (Harbor). Версия чарта и образа может браться из Semver в S3.
- В прод деплой не идёт из Concourse напрямую: образ и версия попадают в **репозиторий манифестов**, откуда их подхватывает Flux.

### GitOps (Flux)

- **Flux** следит за Git-репозиторием с манифестами (HelmReleases, Kustomize или raw YAML по окружениям). После мержа изменений (новый тег образа, обновление values) Flux синхронизирует состояние кластера с Git.
- Concourse лишь обновляет в этом репо нужные поля (тег образа, версия чарта); применение в кластер — зона ответственности Flux. Откат — revert коммита в репо манифестов.

### Сетевая плоскость: вместо стандартного kube-proxy — Cilium

- В кластере Kubernetes вместо штатного **kube-proxy** (iptables/userspace) используется **Cilium** в режиме замены kube-proxy (eBPF для Service/ClusterIP и маршрутизации).
- Плюсы: меньше нагрузки на ноды, лучшая наблюдаемость (Hubble), возможность политик на уровне L3/L4/L7, совместимость с существующими Service/Ingress. Настройка через Helm-чарт Cilium, установка при поднятии кластера или миграции с default CNI.

### Дополнительно

- **Уведомления**: каждая важная джоба шлёт результат в Telegram (успех/провал/отмена) с ссылкой на билд в Concourse.
- **Merge Request**: пайплайн по MR — проверки (lint, build, test), обновление статуса MR (success/failed) и уведомление в чат.
- **Инфра-пайплайны**: обновление нод (Ansible), настройка containerd на группах хостов, при необходимости — каскад обновлений (etcd, S3, concourse, rabbitmq, nodes, gitlab и т.д.) через те же плейбуки из Concourse.

## Инструменты

| Категория | Инструменты |
|-----------|-------------|
| **CI** | Concourse CI (fly CLI, resources, tasks, jobs, set_pipeline) |
| **Конфиг пайплайнов** | Git (репозиторий concourse), YAML-шаблоны по командам/приложениям |
| **Автоматизация хостов** | Ansible (inventory, playbooks, roles), concourse-ansible-playbook-resource |
| **Сборка и образы** | OCI build task, Dockerfile, Harbor (registry) |
| **Версионирование артефактов** | Semver (S3 backend) |
| **Чарты и деплой** | Helm 3 (template, package, push OCI), Helm-тесты в K8s |
| **GitOps (CD)** | Flux (Kustomize/HelmReleases, синхронизация кластера с Git) |
| **Кластер и сеть** | Kubernetes (multi-DC), Cilium (eBPF, замена kube-proxy) |
| **Отказоустойчивость** | Размещение в нескольких ЦОДах, Master/Slave (control plane + workers, PostgreSQL primary/standby) |
| **Интеграции** | GitLab (git, Merge Request resource), Telegram (уведомления), S3 (semver) |

## Результат

- Вся платформа (Concourse, Flux, Harbor, вспомогательные сервисы) работает **на Kubernetes**; разнесение по **нескольким ЦОДам** и топология **Master/Slave** (control plane / воркеры, primary/standby для БД Concourse) обеспечивают отказоустойчивость при сбое дата-центра или узла.
- Один контур: конфиг пайплайнов и плейбуков в Git, обновление пайплайнов через `set_pipeline` по коммиту (GitOps для CI).
- Ansible выполняется из Concourse по расписанию/триггеру — единая точка запуска и аудита для конфигурации серверов и нод.
- Сборка образов и Helm-чартов с линтингом и тестами, публикация в Harbor; деплой в прод только через Flux по изменению репо манифестов.
- Сетевая плоскость на Cilium вместо kube-proxy даёт предсказуемую производительность и возможность развивать наблюдаемость и политики без смены всего стека.
